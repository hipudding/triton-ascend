# 版本策略

## 版本号规则

Triton-Ascend 遵循 [PEP 440](https://peps.python.org/pep-0440/) 版本规范，版本号与上游 Triton 保持对齐：`vMAJOR.MINOR.PATCH[rcN][.postN]`

- **MAJOR.MINOR**：与上游 Triton 版本一一对应，如 Triton-Ascend `3.2` 基于 Triton `3.2`
- **PATCH**：Triton-Ascend 的 `PATCH` 版本可能多于上游 Triton，用于做 `MAJOR.MINOR` 级别的问题修复或改进，例如 Triton-Ascend `3.2.0` 和 `3.2.1` 均是基于 Triton `3.2.0`
- **rcN**：预发布候选版本（Release Candidate），按需发布，供社区提前测试和反馈
- **postN**：已发布版本的问题修复版本（Post Release），按需发布，针对已有稳定版本的补丁

## 版本分支策略

- `main` 分支为最新开发分支，跟踪上游 Triton 最新版本
- 每个发布版本创建对应的版本开发分支（如 `releases/v3.2.x`），该分支与社区拉取 release 的 commit id 相同
- 功能开发应在 fork 仓库中完成，通过 `PR` 合并至 Triton-Ascend 仓库

**`main` 分支对应关系：**

| ------------- | ------------------------------------------------------------ | --------- | ----- | ------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| `main`        | [cfc0a9d](https://github.com/triton-lang/triton-ascend/commit/cfc0a9d) | `3.9~3.13` | `9.0.0` | `2.7.1`   | [fad3272](https://github.com/llvm/llvm-project/commit/fad3272) | [fad3272.patch](https://github.com/triton-lang/triton-ascend/blob/main/third_party/ascend/llvm_patch/fad3272.patch) |

## 维护分支与生命周期

维护分支的状态分为：

- **活跃（`Active`）**：持续接受 Bug 修复、功能改进和安全补丁，会继续特性演进或发布新版本
- **维护（`Maintenance`）**：仅接受关键 Bug 修复和安全补丁，不再发布功能改进
- **归档（`End of Life`）**：不再接受任何修复，分支停止维护

| 分支              | 状态     | 对应 Triton 版本 | Triton-Ascend 发布版本              | 维护截止日期 |
| ----------------- | -------- | ---------------- | ----------------------------------- | ------------ |
| `main`            | `活跃`   | `3.5.0`          | /                                   | /            |
| `releases/v3.2.1` | `活跃`   | `3.2.0`          | `3.2.1`                             | /            |
| `releases/v3.1.x` | `维护`   | `3.2.0`          | `3.2.0rc2`，`3.2.0rc3`，`3.2.0rc4`，`3.2.0` | /            |

## 发布周期

- **稳定版本**：根据项目版本节奏发布，并非每个上游 Triton 版本都发布对应的稳定版本
- **rc 版本**：与上游 Triton 版本节奏统一发布，供早期用户测试
- **post 版本**：按需发布，针对已有稳定版本的问题修复
